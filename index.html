Hospital Records Management System
From Excel to MS Access

I tried to build a hospital patient records system in Excel, hit a wall, figured out why, then built the right solution in Microsoft Access. This page documents that whole journey.

Data Modeling Relational Database Design SQL Queries in Access Excel-to-Database Migration

Where It Started

What I Was Trying to Build

The goal was to build a simple system to manage hospital patient records — things like tracking which patients are admitted, recording doctor visits, calculating bills, and keeping payment status up to date. Think of it like a digital filing system for a small hospital.

My first instinct was to use Microsoft Excel — it's what I knew best, and it seemed like it could handle tables of data. I gave it a serious try. It didn't work out. This page explains what I tried, why each approach broke down, and how I eventually solved it properly using Microsoft Access.

Looking back, there's actually a second layer to why this got so complicated. I wasn't just trying to store records — I also wanted to analyse them in the same place. Things like a "Summary" sheet showing "Today's Visits" or "This Month's Visits", or status labels like "Active OPD" and "Admitted (IPD)" that update automatically based on the raw data. In simple terms: storing data and analysing data are two different jobs. A spreadsheet is great at the second job, but it isn't built for the first one — safely storing and organising raw records with rules. I was trying to make Excel do both jobs in the same sheets, and every time the "storage" side had even a small inconsistency, the "analysis" formulas built on top of it broke. That's the thread running through both attempts below.

Phase 1 — What I Tried in Excel

Two Approaches — Both Unsuccessful

Before switching tools, I genuinely tried to make Excel work. Here are the two different ways I approached it and what went wrong with each one.

1
Putting Everything in One Sheet
One row per patient, all information in one place

My first idea was simple: one big table with everything in it. Each row would be one patient, and the columns would cover their personal details, visit dates, diagnoses, billing amounts, and payment status — all on a single sheet called "Patients Record".

The problem showed up quickly. Some patients — like P-008 and P-014 — first visited as outpatients (OPD) and were later admitted as inpatients (IPD). That means they have two separate billing records. But in my table, there was only one row per patient. What do I put in the "Total Amount" column for them? Which bill do I show?

The deeper issue is that one row can only hold one visit. If a patient comes back three times, do I add three rows? Then I'd have to repeat their name, date of birth, blood group, and address three times — and if any of it changes, I'd have to update every single row manually. It became a mess very fast.

2
The Excel Formula Approach
Splitting into multiple sheets, connecting them with XLOOKUP, and trying to track patient status and bills with formulas

My second attempt looked smarter: I split the data across different sheets — one for patient details ("Patients Record"), one for OPD billing, one for IPD billing, and one for reference values like ward rates and also converted each sheet into a proper Excel Table (using Insert → Table), giving them names like tblPatients, tblOPD, and tblIPD, so I could write cleaner formulas using column names instead of cell references. Then I used Excel's XLOOKUP formula to pull patient names from the central patient sheet into the billing sheets automatically, as shown in the following screenshot.

OPD Billing sheet — the "Patient Name" column uses XLOOKUP to fetch the name from "Patients Record" based on Patient ID.

This part actually worked fine on its own — XLOOKUP successfully pulled names across sheets. But pulling a name is the easy part. What I really wanted was for my "Patients Record" sheet to automatically know where each patient currently stands — were they just registered, currently an OPD patient, or admitted as an IPD patient? So I added a column on Patient Records sheet called "Current Condition" with this formula, as shown in the following screenshot:

=IF(XLOOKUP([@[Patient ID]],tblIPD[Patient id.],tblIPD[Patient id.],"")<>"","Admitted (IPD)",
  IF(XLOOKUP([@[Patient ID]],tblOPD[Patient id.],tblOPD[Patient id.],"")<>"","Active OPD","Registered"))
Patients Record sheet — the "Current Condition" formula checks whether each Patient ID exists in the IPD or OPD tables.

In plain English, this formula checks: "Does this Patient ID show up anywhere in the IPD table? If yes, this patient is 'Admitted (IPD)'. If not, does it show up in the OPD table? If yes, they're 'Active OPD'. Otherwise, they're just 'Registered'." This is essentially me trying to recreate, with a formula, the kind of status check a database query would do automatically.

Then I went one step further. Some patients — like P-008 and P-014 — first came in as OPD patients and were later admitted to IPD. That means their Patient ID exists in both tables at once. So I added a second column on Patient Records sheet, "Condition Transition", with this formula, as shown in the following screenshot:

=IF(AND(XLOOKUP([@[Patient ID]],tblIPD[Patient id.],tblIPD[Patient id.],"")<>"",
       XLOOKUP([@[Patient ID]],tblOPD[Patient id.],tblOPD[Patient id.],"")<>""),
   "OPD → IPD Transition","")
Patients Record sheet — the "Condition Transition" formula flags a patient as "OPD → IPD" when their ID exists in both billing tables.

This one checks: "Does this Patient ID exist in BOTH the IPD table AND the OPD table? If yes, flag this patient as 'OPD → IPD Transition'." At this point I felt like I was actually building something close to a real status-tracking system — just using formulas instead of a database.

But then I noticed a problem: even with this flag set on the "Patients Record" sheet, that same patient's old OPD record was still sitting in the OPD Billing sheet, still looking like an active, unresolved OPD visit. So I went into the OPD Billing sheet and added yet another column, "Current Status", with this formula, as shown in the following screenshot:

=IF(XLOOKUP([@[Patient id.]],tblIPD[Patient id.],tblIPD[Patient id.],"")<>"","OPD → IPD Transition","OPD")
OPD Billing sheet — the "Current status" formula re-checks the same patient against the IPD table from the OPD side.

This says: "From the OPD Billing sheet's point of view — does this same patient also show up in the IPD table now? If yes, mark this OPD record as 'OPD → IPD Transition' too. Otherwise, leave it as 'OPD'."

So now I had three different formulas in three different places, all trying to answer a version of the same question: "what is this patient's current status?" And that's exactly where it fell apart.

Unsuccessful — Status flags updated correctly, but the data sitting next to them didn't follow

The next thing I tried was about closing the loop: I wanted the "Patients Record" sheet to show, at a glance, how much each patient currently owed — without having to open the billing sheets separately. So I added a "Total Amount" column with this formula, which relies on the "Current Condition" column above, as shown in the following screenshot:

=IFERROR(

     XLOOKUP([@[Patient ID]],tblIPD[Patient id.],tblIPD[Net amount],""),
  IF([@[Current Condition]]="Active OPD",
     XLOOKUP([@[Patient ID]],tblOPD[Patient id.],tblOPD[Net amount],""),
     "")
  ),
"")
Patients Record sheet — the "Total Amount" formula pulls the Net Amount from either tblIPD or tblOPD depending on Current Condition.

In plain English: "If this patient's Current Condition is 'Admitted (IPD)', go fetch their Net Amount from the IPD billing table. If it's 'Active OPD' instead, fetch it from the OPD billing table. Otherwise, leave it blank. And if anything at all goes wrong, just show blank instead of an error."

Here's exactly where it broke. For patients like P-008 and P-014, the "Condition Transition" formula above had already correctly flagged them as "OPD → IPD Transition". But the "Total Amount" formula only checks for two exact values: "Admitted (IPD)" or "Active OPD". Their "Current Condition" still technically read "Admitted (IPD)" (because they were now in the IPD table), so the formula did try to pull from tblIPD — but their IPD record was incomplete (still mid-transition), so the lookup returned an error, which IFERROR quietly turned into a blank cell.

The result: no error message, no red flag — just an empty cell that looks like the patient simply hasn't been billed yet. That's arguably worse than a visible error, because nothing tells you to go check.

And there's a second, deeper issue here, even for patients where the formula did work. If a patient ever has two active bills at the same time — one OPD, one IPD — then "the patient's Total Amount" isn't a single number anymore. It's two numbers, for two different visits. The formula can only return one of them, based on whichever condition matches first. So even when it "worked," it was quietly picking one bill and hiding the other.

This is also a design problem, not just a formula problem. In a relational database, a patient's bill isn't a property of the patient — it's a property of a specific visit or admission. Storing "Total Amount" on the master patient record assumes each patient has exactly one bill at a time. The moment that assumption breaks (as it did for P-008 and P-014), the whole column becomes unreliable. This was the moment I realised: this isn't a formula bug I can patch — it's a structural problem with where I'm storing this information.

Unsuccessful — A single "Total Amount" field can't represent a patient with two separate, simultaneous bills

At this point I'd tried multiple formula-based approaches within Excel, and every one of them broke down on the same two patients (P-008 and P-014) for slightly different reasons. That's when I started researching properly — and that's how I found out about Database Management Systems, and eventually landed on Microsoft Access.

The Real Reason

Why Excel Simply Isn't the Right Tool Here

After these attempts, I realised the problem wasn't my Excel skills — it was that I was using the wrong tool entirely. Excel is designed for calculations and analysis on flat data. A database management system (DBMS) is designed for storing and managing connected records with rules. These are genuinely different jobs.

Here are the specific things I needed that Excel just doesn't have:

Gap 1

No way to make a Patient ID truly unique

In a real database, you can mark a field as a "Primary Key" — meaning no two rows can ever have the same value there. Excel has no equivalent. You can type the same Patient ID twice and nothing stops you, which leads to confusing duplicate records.

Gap 2

No protection against "orphan" records

When I deleted a patient from the patient sheet, their billing rows on other sheets still sat there — now pointing to no one. A real database can say "if this patient is deleted, also delete their billing records (or block the deletion entirely)." Excel has no such rule.

Gap 3

XLOOKUP is not the same as a proper table connection

XLOOKUP is more flexible than VLOOKUP, but it has the same fundamental limit: it grabs a snapshot of a value at the moment the formula runs — it doesn't create a live, permanent link between two tables. As soon as a patient existed in both the OPD and IPD tables at once (like P-008 and P-014), my chain of XLOOKUP formulas — "Current Condition", "Condition Transition", "Current Status" — started returning answers that contradicted each other, and the data sitting in nearby columns never updated to match.

Gap 4

One patient can't have multiple visits in one row

A hospital patient might visit ten times. In Excel, each visit would need its own row — which means copying the patient's name, age, blood group, and address ten times. If you update their address once, you have to find and fix all ten rows. A database stores the patient details once and links all their visits to that one record.

Gap 5

Incomplete records cause cascading errors

Patient P-008 was mid-transition from OPD to IPD — their ward type was blank. In Excel, this caused the entire billing row to show #VALUE! errors, which then broke the totals, which then broke other formulas that depended on those totals. A database either saves a complete record or doesn't save it at all — there's no "half-saved" broken state.

Gap 6

Calculated totals don't belong on the master record

My "Total Amount" column tried to store a single bill total directly on each patient's master record. But a bill belongs to a specific visit or admission — not to the patient as a whole. The moment a patient had two active bills at once, "the patient's Total Amount" stopped being a meaningful question. A database keeps facts about the patient (name, DOB, blood group) separate from facts about their visits and bills, and calculates totals on demand instead of storing them.

The Fix

Switched to Microsoft Access

Access is a proper database tool. It handles all six of the above as built-in features — unique IDs, linked tables, protection against missing references, one-to-many relationships, data validation, and calculated fields that live in queries instead of master tables. Once I moved there, all the problems that seemed hard in Excel just... didn't exist.

Phase 2 — The Right Approach

How I Built It in MS Access

In Access, instead of one big spreadsheet, I split the data into separate tables — each one storing one type of information. Then I drew connections between them so Access knows how they relate to each other. This is called a relational database.

Think of it like this: instead of writing a patient's name on every bill, you write their Patient ID — and Access knows to look up the name from the patient table automatically. Change the name once in one place, and it updates everywhere.

This directly fixes the problems from my Excel formula approach. Notice that Name only appears in the "Patients Record" table below — not in "OPD Visits" or "IPD Admissions". Each table only stores the Patient ID (a foreign key), and the name is fetched automatically through that link whenever it's needed. Similarly, Age isn't stored anywhere — it's calculated from the patient's date of birth whenever a query needs it, and No# of days for an IPD stay is calculated from the admission and discharge dates.

Four connections, each enforced by Access: a patient (1) can have many IPD admissions and many OPD visits (∞); a ward type in IPD Rates (1) can apply to many IPD admissions (∞); and a bill type in OPD Rates (1) can apply to many OPD visits (∞). None of these connections existed in Excel — they were just hopeful XLOOKUP formulas pointing in roughly the right direction.

Queries That Replace Broken Formulas

This is really the heart of the difference between Excel and Access. In Excel, I was trying to keep everything in one place — a patient's basic details, their current condition, and their bill total, all in the same "Patients Record" table, using formulas like the ones in Attempts 2 and 3. In Access, the "Patients Record" table only stores facts that genuinely belong to the patient (name, date of birth, gender, address...). Nothing about visits, admissions, or money lives there.

Instead, all of that — ages, stay durations, rates, totals, discounts — is calculated on the fly by a query, which is just a saved question you ask the database. Here's the actual query I built for IPD billing:

SELECT
  [Patients Record].[Patient ID], [Patients Record].Name,
  [IPD Admissions].[Admission ID], [Patients Record].DOB,
  -- Age is calculated here, never stored
  DateDiff("yyyy",[DOB],Date())
    -IIf(Format([DOB],"mmdd")>Format(Date(),"mmdd"),1,0) AS Age,
  [Patients Record].Gender, [Patients Record].[Mobile No],
  [IPD Admissions].[Ward ID], [IPD Admissions].[Ward Name],
  [IPD Admissions].[Bed No#],
  [IPD Admissions].[Admission date], [IPD Admissions].[Discharge date],
  -- Number of days is calculated here, never stored
  [Discharge date]-[Admission date]+1 AS [No# of days],
  [IPD Rates].[Ward Rate per day], [IPD Rates].[Room Charges],
  [IPD Rates].[Nursing Charges], [IPD Rates].[Doctor Charges],
  -- Total Amount is calculated here — the exact thing my Excel formula approach tried to store on the patient record
  ([Ward Rate per day]*[No# of days])+([Room Charges]*[No# of days])
    +([Nursing Charges]*[No# of days])+([Doctor Charges]*[No# of days]) AS [Total Amount],
  [IPD Rates].[Discount Rate],
  [Total Amount]*[Discount Rate] AS [Discount Amount],
  [Total Amount]-[Discount Amount] AS [Net Amount],
  [IPD Admissions].Status, [IPD Admissions].[Payment Method], [IPD Admissions].[Payment date]
FROM [Patients Record]
  INNER JOIN ([IPD Admissions]
    INNER JOIN [IPD Rates] ON [IPD Admissions].[Ward ID] = [IPD Rates].[Ward ID])
  ON [Patients Record].[Patient ID] = [IPD Admissions].[Patient ID];

Look at what this query does compared to my Excel attempts. Age — instead of being typed in by hand and going out of date — is worked out from the patient's date of birth, every time the query runs. No# of days is worked out from the admission and discharge dates. And Total Amount, Discount Amount, and Net Amount — exactly the kind of values I was struggling to store on the "Patients Record" sheet in my Excel formula approach — are all calculated step by step, right here, from the IPD Admissions and IPD Rates tables.

This is the core difference: in Excel, I tried to hold general patient information and bill calculations in the same table, and ran into conflicts the moment a patient had more than one type of bill. In Access, "Patients Record" holds only the general, unchanging facts about a patient. Every bill — and every calculation based on that bill — is handled separately by a query like this one, built specifically for that purpose. A patient can have an IPD query result, an OPD query result, or both, with zero conflict, because neither one is competing for the same cell.

The key takeaway: Excel wasn't unsuccessful because I used it wrong or because my formulas weren't clever enough. It is unsuccessful because the features I needed — like unique IDs, real table connections, and data validation rules — don't exist in a spreadsheet. Those are database features, and you need a database tool to get them.
